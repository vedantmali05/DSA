Q1. Why does the standard middle index formula `(left + right) / 2` cause an overflow in Java when working with large arrays, and how does `left + (right - left) / 2` algebraically resolve this without exceeding the array boundary values?
Ans:
- If the right value is the Integer.MAX_VALUE boundary and adding left value to it may cause overflow the capacity of integrer data type in java.
- `left + (right - left) / 2` first substract the left from the max boundary right and divides the obtained value by 2.
- The result is further added with the left which becomes the middle value of left and right.

1.1: Correct! Adding two large integers can exceed `Integer.MAX_VALUE` (2,147,483,647), causing the value to wrap around to a negative number in Java. By doing `left + (right - left) / 2`, we perform subtraction first (which is safe because both are positive and `right >= left`), divide the difference, and add it back to `left`, ensuring we never exceed the bounds.

Q3. Is Selection Sort stable or unstable? Provide a brief step-by-step trace of a small example array with duplicate elements (e.g., `{4, 4, 1}`) that demonstrates how its relative duplicate order is violated.
Ans:
- Selection sort is unstable, because when the elements are duplicate, their relative order is changed.
- But, I don't understand what does a "relative" order mean?

3.1: Let's address your question first! **"Relative order"** means the order of duplicate elements *in relation to each other* as they originally appeared in the input. If you have two duplicates `4` (let's label them `4_a` and `4_b` where `4_a` is initially before `4_b`), a **stable** sort guarantees that `4_a` will still be before `4_b` after sorting. An **unstable** sort might swap them, so `4_b` ends up before `4_a`.

Here is the step-by-step trace for `{4_a, 4_b, 1}`:
- **Pass 1:** Selection sort scans the array for the minimum element, which is `1` at index 2. It swaps `1` with the element at the first position (index 0), which is `4_a`.
- **Result after Pass 1:** The array becomes `{1, 4_b, 4_a}`.
Notice that the relative order of the duplicates has changed: `4_b` now appears *before* `4_a`! This demonstrates why Selection Sort is unstable.

Ans:
- Bubble sort just swaps the 2 consecutive, only if the first one is larger than the second.

Q4. Explain the three-reversal algorithm to rotate an array of size N to the right by K places. Why is it preferred over auxiliary array approaches in production?
Ans: Reverse the elements in an array from 0 to k-1, then reverse the K to arr.lenght - 1 array elements, then reverse the entire array. All three reversals shall be inplace. What does "auxiliary" mean?

4.1: Let's address your question first! **"Auxiliary"** means *extra* or *supplementary*. In computing, "auxiliary space" refers to the temporary memory allocated by an algorithm *excluding* the input data itself. 
- An **auxiliary array approach** creates a brand-new array of size N to copy elements into (requiring O(N) auxiliary space).
- The **three-reversal approach** does all swaps in-place inside the original array, requiring only O(1) auxiliary space (just a single temporary variable for swapping). In production, minimizing memory allocation prevents memory overhead and garbage collection pauses.

Also, a quick note: the reversal steps you described (`0 to k-1`, then `k to arr.length-1`, then entire array) actually perform a **left** rotation. For a **right** rotation, the steps are:
1. Reverse the last K elements (`arr.length - k` to `arr.length - 1`).
2. Reverse the first `arr.length - k` elements (`0` to `arr.length - k - 1`).
3. Reverse the entire array (`0` to `arr.length - 1`).

🟢 4.2: If we want to rotate `{1, 2, 3, 4, 5}` to the right by K = 2 using the block reversal method, what will the array look like after the first two partial reversals (before the final full reversal)?
Ans: {2, 1, 5, 4, 3}

Q5. In the two-pointer technique for removing duplicates from a sorted array in-place, why does the write pointer only write when `arr[read] != arr[write - 1]`? What logical bug occurs if we check `arr[read] != arr[read - 1]` instead?
Ans:
- The read element is started from 1, and write started from 0th index.
- So, read starts reading, and write follows it.
- The reading position item and writing position items don't match when they are not equal (unique, not duplicate).
- Thus, we increment read pointer first, and thus now read == write
- Then, we read & write on the same position, and items remain unchanged.
- But, when arr[read] == arr[write], duplicate is found, so we don't increment the write pointer, only read is incremented.
- Hence, the read pointer keeps incrementing until all the consecutive duplicates are crossed.
- Once, the read pointer reaches the next unique item,
    - write pointer is incremented: because we don't want to update the first occurrence (first one is not duplicate, at least one occurrence should be kept in the array)
    - The arr[read] is written at the arr[write]
    - Hence, all the duplicate occurrences are replaced with upcoming unique items in the array.
- If arr[read] != arr[read - 1] is checked and write is performed,
    - only the last occurrence of the consecutive duplicate will be replaced with next unique elements

5.1: Excellent! Comparing `arr[read]` with the already-written unique element `arr[write-1]` ensures we correctly screen duplicates against the unique prefix we are building. If we checked `arr[read] != arr[read-1]`, we would only compare elements in the original unswept array, which causes logical errors once we start overwriting elements.

Q8. When reversing an array recursively, what is the auxiliary space complexity in terms of call stack frames? Compare this with the space complexity of the iterative two-pointer array reversal.
Ans:
- When reversing an array recursively, the auxiliary space complexity will be O(n)
- This is because, every time the reversal function calls it self, it allocated memory for its params.
- Thus, if the array is of size N, N/2 comparisons will happen, and a new copy of the array will be pushed to call stack N/2 times
- Ignoring the dominant terms, the space complexity becomes O(n)
- In the inplace array reversal, the array items are compared and swapped on the existing array, without allocating any extra space or creating a new array.

8.1: Great answer! Let's clarify one technical detail: In Java, objects (including arrays) are passed by **reference**. The recursion does not copy the array itself; it just passes the reference. 
However, each recursive call pushes a new **stack frame** (activation record) containing the method parameters (`i`, `j`), local variables, and return address. Since there are N/2 calls active at the deepest point of recursion, this uses O(N) auxiliary stack space.

Q10. In the optimized Bubble Sort, why is the best-case time complexity O(N), and what condition triggers the early exit? Explain why the unoptimized version fails to achieve this.
Ans:
- In optimized Bubble Sort, the array is traversed at least once, before checking the condition of being sorted before early exit.
- Thus, in best case the time complexity is O(n) and it's O(n^2) in avg or best case.
- For the early exit, we maintain a flag, (called isSwapped) and mark it as true, even if one swap happens.
- After the end of one pass, we check if the flag is still false
- If still false, means nothing swapped, hence the array is sorted, and we can break the loop or return the updated array.
- The unoptimized Bubble sort fails to do this early exit because, it doesn't make any explicit checks at the end of each pass, continue onto the next pass until the entire array is repeatedly checked for the order.

10.1: Perfect! The `isSwapped` flag tracks if any elements changed position during a pass. If `isSwapped` remains false after a full pass, the array is already sorted, so we can exit early. The unoptimized version lacks this flag and will blindly run all passes.
